# Tray icon, app icon, auto-update & dev controls

## System tray icon — `TrayIconService`

Runs in both live and demo mode. Solves two problems: the widget windows are
borderless/topmost with no taskbar entry (so a stray Alt+F4 or a fullscreen game
can hide one with no obvious way back), and otherwise the only way to stop the
app would be closing the terminal that launched it.

- Built on `System.Windows.Forms.NotifyIcon` — the `App` project has
  `UseWindowsForms` enabled alongside `UseWPF` for this, because WPF has no
  native tray icon type.
- Icon is `src/IRacingOverlay.App/Assets/app.ico`, loaded at
  `SystemInformation.SmallIconSize` so a scaled display gets the .ico's real
  16/20/24/32px drawings instead of a resampled 32px bitmap. A runtime-drawn
  circle survives as a fallback if the resource can't be loaded, **because the
  tray icon is the only way to quit the app**.
- Context menu: a **checkbox per widget** (Standings, Relative, Fuel, Radar),
  **Dev Controls** (demo mode only), a **UI Scale** submenu (100/125/150/175%),
  **Settings...**, **Check for updates**, **Exit** — plus a **Restart to install
  update** item hidden until an update has been downloaded. Double-click the icon
  shows the Relative.
- The widget items are **checkboxes, not "Show" commands**. A menu that can only
  reveal a widget has no answer to "I don't want the radar"; ticking and
  unticking shows/hides it, and the choice is persisted
  (`OverlaySettings.EnabledWidgets`), so a widget switched off stays off across
  restarts. The demo-only dev panel keeps a plain "show" command — it's
  scaffolding, not a preference.
- **UI Scale** applies a `ScaleTransform` to every overlay window's content root;
  `SizeToContent` then resizes each window to fit, so the whole set scales
  together. It's the *shared* scale — a widget with its own override (set in the
  settings window) ignores it. The active scale is ticked and persisted.
- The menu's checkmarks are re-synced from `SettingsService.Changed`, so toggling
  a widget in the settings window moves the tick here too rather than leaving the
  two surfaces disagreeing.
- The app runs under `ShutdownMode="OnExplicitShutdown"`: closing a widget window
  hides it (`App.HideInsteadOfClose`) rather than destroying it, so the tray's
  items always work. The tray's **Exit** (or any window's right-click **Exit**)
  is the only path that actually ends the process (`App.RequestExit`).
- Windows hides newly created tray icons behind the taskbar's `^` overflow arrow
  by default — expected OS behaviour, not a bug.

**Only one copy per flavour runs at a time** (`SingleInstanceGuard`, a named
mutex claimed in `OnStartup` before any window or settings read). A second launch
exits silently with code 0. Without it, double-clicking the Start-menu entry
while the app was already running gave two tray icons and two stacks of widgets
drawn on top of each other — invisible as a *duplicate* precisely because these
windows have no taskbar entry. The mutex name is **scoped by install kind**, so a
source build and the installed app don't block each other (they don't share a
settings file either — see [settings](settings.md#layout-persistence)). An
`AbandonedMutexException` (previous holder crashed) is treated as "the slot is
free" — refusing to start until reboot would be worse than the duplicate.

## Application icon — `Assets/app.ico`, `tools/MakeAppIcon.ps1`

A dark rounded panel (the widgets' own material) carrying an azure radar ring
around a warm centre dot — the same accent/warm pairing the overlay uses for
"everyone else" versus "this is you", so the icon reads as the same product as
the widgets rather than generic chrome.

- Used by the **installer, Add/Remove Programs, the Start menu, the taskbar, the
  exe in Explorer** (`<ApplicationIcon>` + `vpk pack --icon`), the **tray**, and
  the **settings window** chrome.
- **Committed as a binary but generated by a script**, `tools/MakeAppIcon.ps1`,
  so the design is editable — the palette and geometry are code, not an opaque
  file nobody can change. Regenerate with
  `powershell -ExecutionPolicy Bypass -File tools/MakeAppIcon.ps1`.
- Each size is **drawn at its own size**, not scaled from one bitmap: the 16px
  entry drops the edge hairline and tightens the corner radius, which a
  downscale would turn to mush.
- **Entries at 16–64px must stay classic DIB**; 128/256 are PNG.
  `System.Drawing.Icon` — exactly what `NotifyIcon` uses — cannot decode
  PNG-compressed entries, so an all-PNG icon renders fine in Explorer and throws
  in the tray. The two largest stay PNG because DIB would add ~320KB for no
  visible gain.

## Auto-update — `UpdateService`

Built on Velopack's `UpdateManager`, pointed at the public GitHub Releases feed
the app is published to (`.github/workflows/release.yml`). Public repo, so no
access token.

- On launch — and on demand via the tray's **Check for updates** — it checks the
  feed and, if a newer release exists, **downloads it in the background**. When
  the download is ready it reveals the tray **Restart to install update vX.Y.Z**
  item and shows a balloon notification.
- Applying an update (which restarts the app) only happens when the user clicks
  that item — **never automatically mid-session**, which matters for a racing
  overlay. `App.ApplyUpdate` disposes the tray icon and calls
  `ApplyUpdatesAndRestart`.
- **Installed copies only.** `UpdateManager.IsInstalled` is false under `dotnet
  run` or a portable unzip, so dev/demo launches no-op — no network call, no
  error. Requires the `VelopackApp.Build().Run()` bootstrap in `App.Main`.
- Failures are caught and logged to `%LocalAppData%\IRacingOverlay\update.log`
  and never interrupt a session. A **manual** check does report them:
  `CheckAndDownloadAsync` returns an `UpdateCheckResult` whose
  `UpdateCheckStatus` separates `UpToDate` from `Failed`, and the tray says
  *"Couldn't check for updates"* rather than *"You're on the latest version"*.
  **Those two must never look the same to the user** — they were the same `null`
  return once, which is how a completely broken feed went unnoticed for three
  releases.
- **The repo being public is a load-bearing assumption of the whole update
  path.** Velopack asks for the release feed unauthenticated; against a private
  repo GitHub answers 404, the check fails, and every installed copy silently
  believes it's up to date, with the only evidence buried in `update.log`.
- No automated tests — SDK glue. The feed read was verified end-to-end against a
  live release, which the installed copy downloaded on launch.

## Dev control panel — `DevControlWindow` / `DevControlViewModel` / `IDemoControls`

Shown automatically alongside the other widgets, **only when running with
`--demo`** — it drives `SimulatedTelemetrySource` live, so it has nothing to
control in live mode and doesn't appear there.

| Control | Effect |
|---|---|
| **Cycle race type** | Steps through the demo race presets (`RacePresets`) — IMSA (3-class GTP/LMP2/GTD), the GT3 single-class series, the Porsche Cup single-make, and the two Mazda MX-5 Cups — wrapping around. Each rebuilds the field with that series' classes, colours, per-class pace and typical grid size. IMSA is the default. |
| **+ Add car** / **− Remove** | Grows/shrinks the simulated field, 3–40 cars (`MinCarCount`/`MaxCarCount`), rebuilt deterministically from the active preset for the new size. |
| **− 5 L** / **+ 5 L** | Adjusts player fuel, clamped to a 65 L tank. Adding fuel mid-lap exercises the same refuel-detection path (`FuelCalculator`'s 0.2 L threshold) a real pit stop would. |
| **Set critical (2 L)** | Drops fuel straight to 2 L, to check the fuel widget's red "LAPS SHORT" state without waiting for a real burn-down. |
| **Cycle wetness** | Dry → Very Lightly Wet → Moderately Wet → Very Wet → wraps, to check the relative's wetness badge. |
| **+ Incident** | Increments the player's incident count in the relative session strip. |
| **Toggle player pit** | Flags the player's own row as pitting (surface `InPitStall`), to check the PIT badge and opacity dimming on the player row specifically. |
| **Cycle session** | Practice → Open Qualify → Race → wraps, each with its matching setup file, bumping the session number so the setup reminder's flash re-triggers. Resets the "modified" flag, matching a freshly loaded setup. |
| **Toggle setup modified** | Flags the loaded setup as modified, to check the setup strip's "MOD" tag. |
| **Cycle radar** | Clear → CarLeft → CarRight → CarLeftRight → TwoCarsLeft → TwoCarsRight → wraps, to check the radar's first-lap spotter fallback. Also what `scripts\render.ps1 radar-danger` drives. |

**Implementation rule:** all mutations happen under the same `_gate` lock the
background timer thread uses to read state, since dev-panel clicks (UI thread)
and telemetry generation (timer thread) touch the same fields concurrently.

**This file has no automated tests and a real bug shipped here once.**
`AddCar()` indexed its name/number lookup by `currentFieldSize -
initialFieldSize`, which goes negative — `IndexOutOfRangeException`, crashing the
whole app — once `RemoveCar()` had shrunk the field below its initial size. It
was caught via Windows Event Log forensics after a launched instance crashed
silently. That incremental-roster path has since been replaced: Add/Remove and
Cycle race type all funnel through one deterministic `RebuildField(count)`, so
that class of indexing bug is gone by construction. Before trusting a change
here, stress-test it — see
[DEVELOPMENT.md](../DEVELOPMENT.md#dev-control-panel-demo-mode).
